# Phase 15.1: Dead Code Audit via LSP - Research

**Researched:** 2026-01-23
**Domain:** TypeScript dead code detection, LSP findReferences, static analysis
**Confidence:** HIGH

## Summary

This phase focuses on identifying and removing unused/dead code in the gestao_fronteira Next.js 16 TypeScript codebase using LSP `findReferences` capabilities. The codebase contains 347 TypeScript files across `app/`, `components/`, `lib/`, and `types/` directories.

The research reveals two complementary approaches:

1. **LSP-based manual audit** - Using `mcp__lsp__findReferences` to systematically verify each exported symbol has usages beyond its definition. This is prescriptive and integrates with the existing LSP setup in `.planning/config.json`.

2. **Automated tooling for validation** - Using `knip` (successor to ts-prune) to cross-validate findings. Knip has built-in Next.js plugin support and can detect unused exports, files, and dependencies.

**Primary recommendation:** Use LSP `findReferences` for the audit process as specified in phase requirements, supplemented by `knip` for validation and edge case detection. The LSP approach provides fine-grained control and git-friendly atomic commits.

### Pattern 1: LSP Audit Workflow

**What:** Systematically check each exported symbol for references.
**When to use:** For all non-entry-point exports in lib/, components/, types/.

**Algorithm:**
```
For each file in audit scope:
  1. Get symbols: mcp__lsp__documentSymbol(filePath)
  2. For each exported symbol:
     a. refs = mcp__lsp__findReferences(filePath, symbol.line, symbol.char)
     b. If refs.length == 1 (only definition):
        - Mark as potentially unused
        - Verify not a Next.js special export
     c. If refs.length == 0:
        - Symbol is definitely unused
```

**Example:**
```typescript
// Source: .claude/get-shit-done/references/lsp-patterns.md

// Step 1: Get all symbols in a utility file
documentSymbol("lib/utils/fuzzy-search.ts")
// Returns: [{ name: "fuzzySearch", line: 5, kind: "function", exported: true }]

// Step 2: Check each exported symbol
findReferences("lib/utils/fuzzy-search.ts", 5, 10)
// If returns only the definition location → DEAD CODE
```

### Pattern 2: Batch Verification with Knip

**What:** Run knip to get a comprehensive list of unused exports.
**When to use:** Initial scan and validation of LSP findings.

**Example:**
```bash
# Run knip analysis
npx knip --include exports,files

# Output sample:
# Unused exports:
# lib/utils/fuzzy-search.ts:5:0  fuzzySearch
# lib/api/inep-integration.ts:10:0  fetchINEPData
```

### Anti-Patterns to Avoid

- **Removing Next.js special exports:** Never remove `default` exports from `page.tsx`, `layout.tsx`, `route.ts` files - these are framework entry points
- **Removing re-exported types:** Types may be unused in runtime code but needed for type checking
- **Removing index.ts barrel exports:** These aggregate exports for convenience; check the re-exported symbols, not the barrel
- **Ignoring dynamic imports:** `import()` and `require()` may reference symbols not detectable statically

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parsing exports | Custom regex | LSP documentSymbol | Handles all export syntax variants |
| Finding references | grep "functionName" | LSP findReferences | Understands scope, ignores comments/strings |
| Detecting unused files | Custom file walker | knip | Handles Next.js patterns, dynamic imports |
| Automatic removal | Manual editing | tsr --write | Handles cascading deletions safely |

**Key insight:** TypeScript's findReferences has edge cases with re-exports (`export * from`) that require workarounds. Tools like tsr and knip have already solved these.

## Common Pitfalls

### Pitfall 1: False Positives from Test Files

**What goes wrong:** LSP finds a symbol is "unused" but it's actually used in test files.
**Why it happens:** Test files may be excluded from analysis scope.
**How to avoid:** Include test file patterns in audit scope, OR verify manually before deletion.
**Warning signs:** Symbol names containing "test", "mock", "stub" patterns.

### Pitfall 2: Missing Dynamic Imports

**What goes wrong:** Removing code that's loaded via `import()` or `require()`.
**Why it happens:** Static analysis can't trace string-based dynamic imports.
**How to avoid:** Search for dynamic import patterns before removing.
**Warning signs:** Files in `lib/` with no static imports but exist for a reason.

```bash
# Check for dynamic imports of a file
grep -rn "import.*fuzzy-search\|require.*fuzzy-search" .
```

### Pitfall 3: Next.js Framework Exports

**What goes wrong:** Removing exports Next.js needs (metadata, generateStaticParams, etc.).
**Why it happens:** These are "used" by the framework, not by application code.
**How to avoid:** Never flag exports in `app/` directory files.
**Warning signs:** Special export names: `metadata`, `generateMetadata`, `generateStaticParams`, `revalidate`, `dynamic`, `runtime`.

### Pitfall 4: Type-Only Exports

**What goes wrong:** Removing a type export breaks type checking elsewhere.
**Why it happens:** Type imports (`import type { X }`) may not create runtime references.
**How to avoid:** Run `pnpm typecheck` after each removal batch.
**Warning signs:** Files in `types/` directory, exports of `type` or `interface`.

### Pitfall 5: Barrel File Confusion

**What goes wrong:** Marking a barrel file's re-exports as unused.
**Why it happens:** The barrel file imports and re-exports; the original source appears unused.
**How to avoid:** Check if symbol is re-exported in an index.ts before flagging.
**Warning signs:** `index.ts` files with only `export * from` or `export { X } from` statements.

### Finding Unused Exports via LSP

```typescript
// Source: .claude/get-shit-done/references/lsp-patterns.md

// Dead Code Detection Pattern
for (const symbol of documentSymbol("lib/utils/helpers.ts")) {
  if (symbol.is_exported) {
    const refs = findReferences(file, symbol.line, symbol.char);
    if (refs.length === 1) {  // Only the definition itself
      console.log(`Potentially unused: ${symbol.name}`);
    }
  }
}
```

## Open Questions

Things that couldn't be fully resolved:

1. **LSP performance on 347 files**
   - What we know: LSP findReferences is fast for individual queries
   - What's unclear: Performance when querying thousands of symbols
   - Recommendation: Batch by directory, pause between batches if needed

2. **Re-export chain detection**
   - What we know: LSP can follow definitions; knip handles re-exports
   - What's unclear: How well LSP handles `export * from` chains
   - Recommendation: Cross-validate with knip for barrel files

3. **Next.js 16 specific patterns**
   - What we know: Knip has Next.js plugin support
   - What's unclear: If all Next.js 16 patterns (like new metadata API) are recognized
   - Recommendation: Never flag app/ directory exports; test on a sample file first
