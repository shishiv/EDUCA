# Phase 7.1: Admin School Selector - Research

**Researched:** 2026-01-19
**Domain:** React Context, Session State Management, UI Components
**Confidence:** HIGH

## Summary

This phase implements a school selector for admin/multi-school users to access escola-scoped data. The research focuses on existing patterns in the codebase - how authentication works, how escola data is fetched, where the selector should integrate, and which pages need updates.

**Key findings:**
1. The `useAuth` hook already provides `userProfile` with `escola_id` and `tipo_usuario` - this is the foundation
2. The app already has React Context patterns (SearchContext, SessionRealtimeContext) that can be followed
3. Existing shadcn components (Command, Popover, Select) provide the UI building blocks for a combobox
4. Multiple pages already filter by `escola_id` - they need to read from the new context instead

**Primary recommendation:** Create an `EscolaContext` using React Context pattern (following existing SearchContext pattern), integrate into dashboard layout, and update escola-scoped pages to read from context.

### Pattern 2: SessionStorage Persistence

**What:** Persist escola selection only in session (clears on logout/tab close)
**Source:** Search preferences in search-context.tsx use localStorage - we use sessionStorage

```typescript
// On selection change
useEffect(() => {
  if (selectedEscolaId) {
    sessionStorage.setItem('gestao-fronteira-selected-escola', selectedEscolaId)
  } else {
    sessionStorage.removeItem('gestao-fronteira-selected-escola')
  }
}, [selectedEscolaId])

// On mount - restore from session
useEffect(() => {
  const stored = sessionStorage.getItem('gestao-fronteira-selected-escola')
  if (stored) {
    setSelectedEscolaId(stored)
  }
}, [])
```

### Pattern 3: Role-Based Visibility

**What:** Determine selector visibility based on user profile
**Source:** `lib/auth.ts` (lines 181-205), sidebar.tsx filtering

```typescript
function shouldShowSelector(userProfile: UserProfile, escolasCount: number): boolean {
  // Admin and Gestor SME always see selector
  if (['admin', 'gestor_sme'].includes(userProfile.tipo_usuario)) {
    return true
  }

  // Diretor and Secretario never see selector (single school)
  if (['diretor', 'secretario'].includes(userProfile.tipo_usuario)) {
    return false
  }

  // Coordenador and Professor see if they have multiple schools
  return escolasCount > 1
}
```

### Anti-Patterns to Avoid

- **Don't put escola_id in URL**: Per CONTEXT.md, keep URLs simple (/turmas not /escolas/:id/turmas)
- **Don't default to first escola**: Force explicit selection (requirement from user)
- **Don't mix sessionStorage and localStorage**: Use only sessionStorage per requirements
- **Don't reload entire page on escola change**: Update data via context, let React handle re-render

## Common Pitfalls

### Pitfall 1: Forgetting Hydration Mismatch

**What goes wrong:** sessionStorage read on server causes hydration error
**Why it happens:** sessionStorage not available during SSR
**How to avoid:** Only read sessionStorage in useEffect (client-side)
```typescript
// WRONG - causes hydration error
const [selected, setSelected] = useState(sessionStorage.getItem('key'))

// RIGHT - defer to client
const [selected, setSelected] = useState<string | null>(null)
useEffect(() => {
  setSelected(sessionStorage.getItem('key'))
}, [])
```
**Warning signs:** "Text content does not match server-rendered HTML"

### Pitfall 2: Race Condition with Auth Loading

**What goes wrong:** Context tries to fetch escolas before user is authenticated
**Why it happens:** Auth is async, context mounts immediately
**How to avoid:** Guard escola fetching on userProfile being available
```typescript
useEffect(() => {
  if (!userProfile) return  // Don't fetch until auth is ready
  fetchEscolas()
}, [userProfile])
```
**Warning signs:** Empty escola list even for admin users

### Pitfall 3: Not Clearing Selection on Logout

**What goes wrong:** Previous user's escola selection persists for next user
**Why it happens:** sessionStorage persists until tab closes
**How to avoid:** Clear sessionStorage in signOut handler
```typescript
// In auth.ts signOut or EscolaProvider
const handleSignOut = () => {
  sessionStorage.removeItem('gestao-fronteira-selected-escola')
  // ... rest of signOut
}
```
**Warning signs:** User sees data from wrong school after re-login

### Pitfall 4: Multiple Context Providers in Wrong Order

**What goes wrong:** EscolaProvider doesn't have access to user data
**Why it happens:** Provider order matters - EscolaProvider needs useAuth
**How to avoid:** Ensure EscolaProvider is inside AuthGuard but can access auth
```typescript
// In dashboard layout
<AuthGuard>
  <EscolaProvider> {/* Has access to useAuth */}
    <SessionRealtimeProvider>
      {children}
    </SessionRealtimeProvider>
  </EscolaProvider>
</AuthGuard>
```
**Warning signs:** "useAuth must be used within AuthGuard" errors

async function fetchEscolas(): Promise<Escola[]> {
  const { data, error } = await supabase
    .from('escolas')
    .select('id, nome')
    .eq('ativo', true)
    .order('nome')

  if (error) throw error
  return data || []
}

export default function TurmasPage() {

## State Management Recommendation

**Recommendation: React Context + sessionStorage**

Rationale:
1. **Not Zustand:** App already uses React Context for similar purposes (SearchContext, SessionRealtimeContext). Adding Zustand for one feature would introduce inconsistency.
2. **Not URL state:** User explicitly decided against /escolas/:id/turmas pattern
3. **sessionStorage over localStorage:** Requirement is session-only persistence

Context shape:
```typescript
interface EscolaContextType {
  // Data
  escolas: Escola[]
  selectedEscolaId: string | null
  selectedEscola: Escola | null

  // Loading states
  loading: boolean

  // Actions
  selectEscola: (id: string | null) => void
  clearSelection: () => void

  // Computed
  shouldShowSelector: boolean
}
```
